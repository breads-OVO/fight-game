## 1. 项目概述

### 1.1 项目背景

基于 Java 版分布式卡牌游戏服务器 `card-game-server` 的架构经验，以 **Go-Zero 微服务框架** 为底座，构建实时格斗游戏服务器。Java 版验证了微服务 + Actor 模型 + gRPC/Protobuf + etcd 服务发现的架构可行性，Go 版在此基础上：

- **框架升级**：使用 Go-Zero 替代裸 Go，利用 goctl 工具链提升工程化效率
- **实时性优化**：回合制卡牌（秒级）→ 实时格斗（毫秒级）
- **架构继承**：微服务拆分、etcd 服务发现、Protobuf 协议设计思路直接复用

### 1.2 项目目标

1. 实现 2 名玩家实时对战的格斗游戏服务器
2. 先采用**状态同步**方案跑通完整链路，后续可选帧同步
3. 以 Go-Zero 微服务框架为底座，构建 etcd + gRPC 微服务架构
4. 支持匹配、观战、断线重连等基础功能
5. 采用 **"服务端先行、模拟客户端驱动"** 的开发策略——先完成服务端核心逻辑和 gRPC 接口，再用命令行/测试工具模拟客户端行为验证

### 1.3 适用范围

本文档覆盖 Go 版格斗游戏服务器 v1.0 所有功能模块的需求定义，适用于开发、测试、运维团队。

## 2. 技术选型

### 2.1 技术栈总览

| 技术 | 用途 | 选型理由 |
|------|------|---------|
| **Go-Zero** | 微服务框架 | 工程化程度高，goctl 工具链提升开发效率；内置服务治理（熔断、限流、超时）；与 etcd + gRPC 生态集成良好；已在游戏行业验证 |
| **Go 1.22+** | 开发语言 | goroutine 轻量，适合实时游戏房间并发模型 |
| **etcd** | 服务注册/发现 | 与 Go-Zero、gRPC 生态集成最佳；分布式、高可用，满足动态扩缩容 |
| **gRPC (Protobuf)** | 服务间通信 | 高性能、强类型；Protobuf 同时用于 WebSocket 客户端通信协议，实现协议统一 |
| **Go-Zero api + gorilla/websocket** | 网关 WebSocket | go-zero 的 api 服务处理 HTTP 升级到 WebSocket，集成 JWT 鉴权、流量控制 |
| **MySQL 8.0 (GORM)** | 持久化 | 存储账号、战绩、Replay 等结构化数据；Go-Zero 官方支持 GORM |
| **Redis 6.0** | 缓存/匹配队列 | 会话缓存、排行榜；Sorted Set 实现匹配队列 |
| **Prometheus + Jaeger** | 可观测性 | Go-Zero 内置 Prometheus 集成；Jaeger 提供分布式链路追踪 |

### 2.2 Go-Zero 框架优势在本项目的体现

| 优势 | 对应项目需求 |
|------|-------------|
| **goctl 代码生成** | 一键生成 gRPC 服务端/客户端代码、API 路由、数据库 model |
| **内置服务治理** | 熔断保护（防止雪崩）、限流（防止恶意请求）、超时控制 |
| **etcd 集成** | 服务注册/发现开箱即用，无需手动编写 EtcdServiceRegistry |
| **Prometheus 指标** | 内置 QPS、延迟、错误率监控，无需额外集成 |
| **错误处理** | 确定性的 error 处理哲学，避免 try-catch 的不可控性 |

### 2.3 开发策略："服务端先行、模拟客户端驱动"

**核心理念**：先完成后端核心逻辑和所有 gRPC 接口定义，再通过模拟客户端验证，最后接入真实客户端。

```
阶段 1: 服务端核心（纯后端，无 UI）
  定义 .proto → goctl 生成 gRPC 服务骨架
  → 实现 Auth / Match / Game 核心逻辑
  → 单元测试 + go test 验证

阶段 2: 模拟客户端驱动（CLI 工具模拟）
  编写模拟客户端（CLI 程序）
  → 模拟 WebSocket 连接、发送输入、接收状态
  → 在此阶段调通完整链路：登录 → 匹配 → 进房间 → 战斗 → 结算

阶段 3: 真实客户端对接
  接入 Unity / Cocos / Web 前端
  → 真实渲染 + 操作输入
  → 调优帧同步体验
```

**为什么这样设计？**

- 格斗游戏的核心复杂度在 **服务端（碰撞检测、伤害结算、帧同步）**，必须先验证服务端逻辑正确
- 模拟客户端可以精准控制输入序列，便于测试边界情况（同时按攻击、断线重连、超时等）
- 前后端可并行开发：后端专注逻辑正确性，前端专注渲染表现

### 3.3 与 Java 版架构对比

| 维度 | Java 卡牌版 | Go 格斗版 (Go-Zero) | 变更原因 |
|------|------------|--------------------|----------|

| 服务发现 | 手动编写 EtcdServiceRegistry | **Go-Zero 内置** | go-zero 与 etcd 集成开箱即用 |

| 服务间通信 | gRPC (手动管理 Stub) | **go-zero rpc（自动生成 Stub）** | goctl 一键生成 |
| 监控 | 无 | **Prometheus 内置 + Jaeger** | go-zero 原生集成 |
| 房间并发 | SingleThreadExecutor Actor | **goroutine-per-room** | goroutine 成本更低 |
| 状态同步 | 完整快照轮询 | 增量状态推送 | 减少带宽，降低延迟 |

## 4. 格斗游戏核心设计

### 4.1 游戏房间 Actor 模型（goroutine + channel）

```
Gateway(WebSocket)
  ↓  消息 → Game 服务
FightRoom goroutine（独立 goroutine）
  │  inputCh  chan PlayerInput     ← Actor 信箱
  │  ticker   *time.Ticker          ← 16ms 固定帧率
  │  frame    int64                 ← 当前帧号
  │  state    *GameState            ← 游戏状态
  │  done     chan struct{}         ← 退出信号
  ▼
  select {
    case input := <-inputCh:    处理玩家输入
    case <-ticker.C:            帧推进 + 广播状态
    case <-done:                清理退出
  }
```

核心骨架（Go-Zero 风格）：

```go
type FightRoom struct {
    Id        string
    Players   [2]*FightPlayer
    InputCh   chan *pb.PlayerInput   // Actor 信箱
    ticker    *time.Ticker
    Frame     int64
    State     *GameState
    done      chan struct{}
    Logger    logx.Logger
}

func (r *FightRoom) Run() {
    r.ticker = time.NewTicker(time.Second / 60)
    defer r.ticker.Stop()
    for {
        select {
        case input := <-r.InputCh:
            r.processInput(input)
        case <-r.ticker.C:
            r.tick()
            r.broadcastState()
        case <-r.done:
            return
        }
    }
}
```

### 4.2 同步方案选择

| 方案 | 复杂度 | 带宽 | 反作弊 | v1.0 选择 |
|------|--------|------|--------|----------|
| **状态同步** | 低 | 中 | 服务端权威 | ✅ v1.0 先用 |
| 帧同步 | 高 | 低 | 需额外校验 | ⏳ v2.0 可选 |

**状态同步方案**：
- 服务端每帧计算完整游戏状态，向客户端推送增量变化（位置、血量、技能 CD）
- 客户端负责渲染，服务端做所有判定

```
    └── 非防御 → 全额伤害 + 击退 + 连击计数
```

## 5. 功能需求

### 5.1 Gateway 网关服务

#### 5.1.1 功能列表

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| GW-001 | WebSocket 连接管理 | P0 | Go-Zero api 服务处理 HTTP → WebSocket 升级 |
| GW-002 | JWT 鉴权 | P0 | 连接建立时验证 Token，go-zero 中间件集成 |
| GW-003 | 消息路由 | P0 | WSMessage.msgType 前缀分发到对应后端 gRPC 服务 |
| GW-004 | 心跳检测 | P0 | 客户端每 10 秒 Ping，服务端 Pong |
| GW-005 | 会话管理 | P0 | PlayerID ↔ WebSocket Conn 映射 |
| GW-006 | 断线检测 | P0 | 30 秒无消息通知 Game 服务 |
| GW-007 | 消息广播 | P1 | 向房间内所有玩家推送帧状态 |
| GW-008 | 流量控制 | P1 | Go-Zero 内置限流器，防止恶意高频请求 |

#### 5.1.2 消息格式

```protobuf
// WebSocket 统一消息包装
message WSMessage {
  int32 msgType = 1;           // 消息类型（对应路由）
  bytes body = 2;              // Protobuf body
  string playerId = 3;         // 发送者
  int64 timestamp = 4;         // 客户端时间戳
  int32 frameNo = 5;           // 帧号（战斗场景）
}

// 通用响应
message WSResponse {
  int32 code = 1;              // 0=成功, 非0=错误码
  string message = 2;
  bytes data = 3;              // 具体业务数据
}
```

#### 5.1.3 Gateway 消息处理流程

```
客户端 WebSocket 连接
  ↓  JWT 中间件验证 Token
  ↓  通过 → 建立会话
  ↓
WSMessage(msgType, body)
  ↓
msgType 路由：
  MSG_AUTH        → Gateway 本地处理（或转发 Auth gRPC）
  MSG_MATCH       → Match gRPC 客户端
  MSG_GAME_INPUT  → 转发到 Game 服务（战斗输入）
  MSG_HEARTBEAT   → Gateway 本地回复 Pong
  ↓
响应通过 WebSocket Conn 写回客户端
帧状态通过 Game 服务推送 → Gateway → WebSocket 广播
```

### 5.3 Match 匹配服务

| 编号 | RPC | 说明 |
|------|-----|------|
| MATCH-001 | `Enqueue` | 加入匹配队列 |
| MATCH-002 | `Dequeue` | 取消匹配 |
| MATCH-003 | `GetStatus` | 查询匹配状态 |

匹配算法：

```
玩家 Enqueue
  ↓  Redis SortedSet: match:queue   score=ELO
Match 定时扫描（每 1 秒，go-zero 的定时任务）
  ↓  ±200 分 → 逐步扩圈至 ±1000
找到 2 人？
  ├── 是：
  │    ├── 从队列移除
  │    ├── Game.CreateRoom() → 返回 roomId
  │    ├── Gateway 推送匹配成功（含 roomId）
  │    └── 玩家 WebSocket 关联到 roomId
  └── 否：等待下一轮扫描
```

```protobuf
service MatchService {
  rpc Enqueue(EnqueueReq) returns (EnqueueResp);
  rpc Dequeue(DequeueReq) returns (DequeueResp);
  rpc GetStatus(GetStatusReq) returns (GetStatusResp);
}

message EnqueueReq {
  string playerId = 1;
  int32 rating = 2;       // 当前段位分
}
```

### 5.4 Game 战斗服务

#### 5.4.1 功能列表

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| GAME-001 | 战斗房间创建/销毁 | P0 | Match 调用 CreateRoom，对局结束销毁 |
| GAME-002 | goroutine-per-room 循环 | P0 | 每个房间独立 goroutine + 16ms ticker |

| GAME-007 | 观战模式 | P1 | 第三方可订阅房间状态 |
| GAME-008 | 断线重连 | P1 | 30 秒内重连恢复对局 |
| GAME-009 | 帧录制 | P2 | 环形缓冲区存储帧输入流，用于回放 |

#### 5.4.2 gRPC 服务定义

```protobuf
service GameService {
  // 匹配成功后由 Match 服务调用
  rpc CreateRoom(CreateRoomReq) returns (CreateRoomResp);
  // 玩家断线后重连
  rpc Reconnect(ReconnectReq) returns (ReconnectResp);
  // 获取房间状态（观战/重连）
  rpc GetRoomState(GetRoomStateReq) returns (GetRoomStateResp);
  // Gateway 转发玩家输入
  rpc SubmitInput(stream PlayerInput) returns (SubmitInputResp);
}
```

#### 5.4.3 帧循环流程

```go
func (r *FightRoom) tick() {
    r.frame++

    // 1. 处理输入队列（当前帧收到的玩家操作）
    r.drainInputs()

    // 2. 更新玩家状态
    r.updatePlayers()

    // 3. 碰撞检测
    r.checkCollisions()

    // 4. 伤害结算
    r.resolveDamage()

    // 5. 检测胜负
    if r.checkKO() {
        r.handleKO()
        return
    }

    // 6. 构建增量状态快照
    snapshot := r.buildSnapshot()

    // 7. 通过 Gateway 推送（gRPC 调用 Gateway.PushToRoom）
    r.gatewayClient.PushToRoom(&pb.PushToRoomReq{
        RoomId:    r.Id,
        FrameNo:   r.frame,
        Snapshot:  snapshot,
    })

    // 8. 记录帧到环形缓冲区（重连/回放用）
    r.recordFrame()
}
```

#### 5.4.4 核心 Protocol Buffers

```protobuf
// 增量状态推送（每帧）
message FrameState {
  int32 frameNo = 1;
  repeated PlayerStateSnapshot players = 2;
  int64 timestamp = 3;
}

message PlayerStateSnapshot {
  string playerId = 1;
  float x = 2;
  float y = 3;
  int32 hp = 4;
  int32 maxHp = 5;
  ActionState actionState = 6;
  bool facingRight = 7;
  int32 comboCount = 8;
  repeated SkillState skills = 9;
}
```

## 12. v1.0 边界与 v2.0 规划

### v1.0 不做（已知限制）

| 编号 | 内容 | 原因 |
|------|------|------|
| LIMIT-1 | 帧同步（仅用状态同步） | 开发成本高，v1.0 先验证完整链路 |
| LIMIT-2 | 3D 格斗（仅用 2D 平面） | 2D 碰撞检测简单，快速验证 |
| LIMIT-3 | 房间状态持久化与迁移 | 宕机后房间丢失，v2.0 做快照恢复 |
| LIMIT-4 | 反作弊校验 | v1.0 服务端权威判定，不做客户端校验 |
| LIMIT-5 | 移动端适配 | 仅支持键盘输入 |
| LIMIT-6 | 观战延迟优化 | 观战者与玩家接收相同帧数据 |
| LIMIT-7 | 完整回放播放器 | 仅录制输入帧，不做可视化播放器 |

### v2.0 规划

```
v2.0 目标：
  ├── 帧同步模式（可选，低带宽）
  ├── 2v2 组队战
  ├── 角色皮肤/装备系统
  ├── 赛季排行榜（Redis Sorted Set）
  ├── 房间快照 + 异地恢复
  └── 观战延迟（30 秒）
```

## 13. 术语表

| 术语 | 说明 |
|------|------|
| Go-Zero | 基于 Go 的微服务框架，内置服务治理、goctl 代码生成工具 |
| goctl | Go-Zero 配套的代码生成工具，一键生成 rpc、api、model 代码 |
| Tick | 一帧，游戏循环的最小时间单位（16.67ms @60fps） |
| State Sync | 状态同步，服务端计算完整逻辑，客户端只负责渲染 |
| Hitbox | 攻击判定框，攻击技能的有效范围 |
| Hurtbox | 受击判定框，角色可以被攻击命中的范围 |
| Actor | goroutine + channel 实现的房间并发模型 |
| Ring Buffer | 环形缓冲区，用于存储最近 N 帧的输入数据 |
| I-Frame | 无敌帧，受击后的短暂无敌时间 |
| Service Mesh | 服务网格，Go-Zero 通过 etcd + gRPC 实现服务间治理 |
